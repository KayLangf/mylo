# CLAUDE.md — Rules for working on Mylo

## Project Overview
Mylo is a proof-of-concept text-based AI eligibility agent for NC FNS (SNAP), built as a case study for CivicReach's Founding Product Engineer role. See SPEC.md for full architecture and scope decisions — SPEC.md is locked; don't deviate from it without flagging the change explicitly.

## Key Commands
- Run agent: `python src/cli.py`
- Ingest/re-embed knowledge base: `python src/ingest.py`
- Run persona tests: `python tests/run_personas.py` (if built — see P2 in SPEC.md)

## Architecture & Folder Structure
See SPEC.md Section 4. Keep this structure — don't reorganize mid-build.

## Coding Conventions
- **Eligibility logic is deterministic code, never LLM-guessed math.** Income thresholds and household-size calculations live in `src/eligibility.py` as plain Python logic, not prompted to the model. This is a hard rule, not a preference.
- **Retrieved knowledge base content is evidence to be cited, not gospel to be repeated verbatim.** System prompts should frame retrieved content as something the model evaluates and grounds answers in, not copies directly.
- **Every chunk in the knowledge base carries metadata**: source document, section, and date. This is required for groundedness checks and citations — don't skip it to save time.
- **Guardrails run before generation, not after.** Crisis detection and injection detection should screen input before it reaches the main conversational LLM call, not filter output after the fact.
- Keep functions small and single-purpose — this is a demo that needs to be walked through live in the review meeting, so readability matters more than cleverness.

## Error Handling
- If retrieval returns nothing relevant, the agent should say so honestly ("I don't have information on that") rather than guessing.
- If the eligibility calculation can't be completed (missing required info), the agent should ask for what's missing rather than assuming.

## Testing Requirements
- Every guardrail (crisis detection, injection resistance, PII handling, out-of-scope refusal) needs at least one manual test case in `tests/personas/` with a documented expected behavior.
- Test the happy path end-to-end before building guardrails on top of it.

## What NOT to do
- Don't add hybrid retrieval, reranking, or multi-language support — explicitly out of scope per SPEC.md.
- Don't build a web UI unless there's clear time remaining after P0/P1 — CLI is the committed interface.
- Don't scrape live websites or attempt to contact real agency systems — case study explicitly prohibits this.
- Don't over-engineer the eval harness — manual persona testing is the committed approach, not automated regression infrastructure.
- Don't build rate limiting or SMS support — documented as future considerations in SPEC.md Section 7, not implemented.

## Critical Design Rule: Session Isolation
Never store conversation state in global or module-level variables. Every session must have its own isolated state object, keyed by session ID. This matters even in a POC — the review meeting is live and concurrent sessions are plausible. Cross-session data leakage is a correctness bug, not a missing feature. See SPEC.md Section 10.

## Learned Rules

- **Chroma metadata truthiness, not key-presence:** When checking the `superseded` field on a retrieved chunk, use `meta.get("superseded")` (truthiness check), never `"superseded" in meta`. Chroma fills omitted metadata keys with `None` for chunks that never had the key written, when other chunks in the same get/query batch do have that key set. A chunk without a `superseded` key will still show up as present-with-value-None on read, which breaks presence checks but not truthiness checks. Verified during Hour 2 ingestion testing (`01_fns_overview.md` chunks, which have no version-conflict blockquote, surfaced as `superseded=None` rather than the key being absent).

- **Retrieval concurrency validated, session-state concurrency now also validated:** Two separate OS processes querying ChromaDB simultaneously (via independent `PersistentClient` instances against the same store) showed zero cross-contamination — confirms the stateless read path is safe to share across concurrent sessions. Session-state concurrency (SPEC.md Section 10) has since been tested directly: two `agent.py` `Session` instances driven through real concurrent `send_message()` calls (two threads, live Anthropic + ChromaDB calls in flight simultaneously) showed zero fact/history leakage between sessions — `Session` isolation is confirmed, not just assumed from the retrieval-layer result. Don't conflate "retrieval layer is concurrency-safe" with "conversation state is concurrency-safe" — they're different claims, but both are now tested.

- **ChromaDB `PersistentClient` concurrent first-init race:** Instantiating `chromadb.PersistentClient(path=...)` against the same on-disk path from two threads at the same time, when no client has been created yet in-process, can crash with `AttributeError: 'RustBindingsAPI' object has no attribute 'bindings'`, surfacing to callers as `ValueError: Could not connect to tenant default_tenant. Are you sure it exists?`. Reproduced via two threads both calling `agent.send_message()` for independent sessions simultaneously with no prior ChromaDB call in-process — one call crashed, the other succeeded. A single warm-up call before spawning concurrent work avoided it, confirming this is specifically a first-init race, not a per-query one.
  - **Fix (in `retrieval.py`):** `_get_collection()` now creates the client/collection handle once behind a `threading.Lock` using double-checked locking, then reuses that handle for every subsequent call. Verified fixed by reproducing the exact crashing scenario (two threads, no warm-up, first ChromaDB access of the process) 3x with no failures, both directly against `retrieval.retrieve()` and end-to-end through `agent.send_message()`.
  - This is a cached *shared read-only resource handle* (client/collection), not per-conversation state — it holds no session history/facts, so it does not violate the Session Isolation design rule, which governs per-conversation mutable data specifically.

- **Retrieval-process narration bug (found and fixed):** The system prompt's groundedness instruction originally caused the agent to describe its own RAG mechanics when expressing uncertainty — phrases like "I don't have that in my retrieved evidence," "in what I've retrieved," and "this turn" leaked into user-facing responses. This is a distinct failure mode from groundedness itself: the *refusal to guess* was working correctly, but the *way it was expressed* exposed internal architecture instead of sounding like a person who doesn't know something.
  - **Fix:** Added an explicit system prompt instruction to state substantive uncertainty directly ("I don't have the exact income limit for a household of six on hand") without referencing retrieval, evidence, or turn-based framing. Citation behavior (source document + effective date) was explicitly preserved and verified unaffected.
  - **Verified via before/after testing:** 5 test questions run against both the original and fixed prompts. All "before" responses contained retrieval-process language; all "after" responses eliminated it while maintaining or improving substantive content and correct source citation. Regression check (household-of-3 income limit query) confirmed citation discipline was fully intact after the fix — sources and effective dates still cited correctly, superseded-document distinction still correctly noted.
  - **General principle for future prompt work:** when fixing a "sounds robotic" complaint, isolate the *specific* mechanical pattern (here: narrating retrieval-as-process) rather than issuing a vague "be more conversational" instruction, which risks degrading unrelated good behaviors like citation discipline. Scope the fix to the exact bug; verify good behaviors are preserved, not just that the bug is gone.

- **`max_tokens` must budget for hidden reasoning, not just visible reply length:** `agent.py`'s Anthropic call was capped at `MAX_TOKENS = 1024`, and multi-part/nuanced user questions were coming back truncated mid-sentence — visible in both a raw REPL `repr()` and in live `cli.py` output, which initially looked like a terminal/display truncation issue rather than an API one. It wasn't: `response.stop_reason` was `max_tokens`, and `response.usage.thinking_tokens` was 530 out of the 1024-token cap — over half the budget was consumed by internal reasoning before any visible reply text was generated, leaving too little room for the actual answer. Confirmed by reproducing the exact failing question and inspecting `response.usage` directly.
  - **Fix:** raised `MAX_TOKENS` to 4096 in `agent.py`. Verified fixed by re-running the same reproducing question end-to-end through `send_message()` — full two-part answer, ends on a proper follow-up question instead of cutting off mid-sentence.
  - When a response looks cut short, check `response.stop_reason` and `response.usage` (especially `thinking_tokens`) before assuming it's a client-side/display truncation — the two look identical from the outside (text just stops), but only one is fixable by raising `max_tokens`.
  - **RECURRED after two rounds of system prompt growth (the ambiguity priority-ordering paragraph + the expanded groundedness paragraph) — a recurring failure pattern tied to cumulative prompt length, not a one-time bug.** A test case (household 4, $3,000/month) came back completely empty (zero text, not truncated) at `MAX_TOKENS=4096`; `stop_reason=max_tokens`, `thinking_tokens=4094` of 4096 — nearly the whole budget spent on reasoning before any output began. Note `thinking_tokens` lives at `response.usage.output_tokens_details.thinking_tokens`, not `response.usage.thinking_tokens` directly — the flat attribute doesn't exist on the `Usage` object.
  - **Fix:** raised `MAX_TOKENS` to 8192, sized for margin against a third round of prompt growth. Spot-checked `thinking_tokens` on other questions (as low as 0, rarely above ~1000) confirming this specific question shape is a genuine reasoning-load outlier, not representative load.
  - **First closure attempt — added smart, signature-specific recovery:** `_is_reasoning_exhausted(response, reply_text)` detects the specific signature (`stop_reason=="max_tokens"`, no tool_use block, no text). `_call_and_resolve_tool_use()` factors out "call once, resolve one round of tool-use-then-continuation" so both the original and a retry can share it. On detection, `send_message()` retries once with `BRIEF_REASONING_NUDGE` appended to the system prompt ("keep your internal reasoning brief... prioritize reaching a visible answer"). Verified via live 20-conversation testing (10x household-4/$3,000, 10x household-8/$8,500): retry fired naturally in 6/20, and all 6 recovered — 0/20 empty. Verified via deterministic mocks: retry-succeeds, both-calls-exhaust-so-fallback-used, and full `send_message()` end-to-end recovery.
  - **Gap found during pre-push regression testing (not closed by the first attempt):** 1 of 20 fresh conversations (household-8/$8,500) still returned a completely empty reply, with the hardcoded `FALLBACK_RESPONSE` never substituted. Root cause, confirmed precisely via mock scenarios rather than assumed: `_is_reasoning_exhausted()` only fires on `stop_reason=="max_tokens"` — but `_call_and_resolve_tool_use()`'s internal tool-result continuation call can independently come back with `stop_reason=="tool_use"` again (the model calling a tool a *second* time with still no text) rather than `"max_tokens"`, and the function only handles one round of continuation. This case is invisible to the signature-specific check and had no fallback protection at all. **Notably, the task's own initial hypothesis for the mechanism (the continuation call itself hitting `max_tokens`) turned out to already be safe** — traced via a dedicated mock (Scenario 4a) confirming `_is_reasoning_exhausted` already catches that sub-case, since it evaluates whichever response `_call_and_resolve_tool_use` last returned, regardless of which of its internal calls produced it. The actual undetected gap was the *second-tool_use* sub-case (Scenario 4b), confirmed via a separate mock that a signature-specific check inherently cannot generalize to.
  - **Final fix — added a dumb, unconditional backstop alongside the smart one:** immediately before `send_message()` appends to history and returns, `if not reply_text: reply_text = FALLBACK_RESPONSE` — unconditional on *why* `reply_text` is empty, so it catches the signature-specific case, the newly-found second-tool_use case, and any future cause not yet discovered. The signature-specific check still exists and still triggers the smarter nudged-retry recovery first; the backstop is purely a final safety net, not a replacement.
  - **Verified:** a baseline reproduction attempt against the *unfixed* code (20 fresh conversations, same two question shapes) came back 0/20 empty this time — consistent with the underlying event being rare (the original occurrence was already only 1/20) and reported honestly as such rather than treated as evidence the bug wasn't real. The mock suite (Scenarios 4a/4b/4c) gives deterministic, on-demand confirmation the fix works, independent of live sampling luck. Post-fix live re-run (20 fresh conversations): 0/20 empty, retry triggered 6/20, and — for the first time — the hardcoded fallback was actually observed firing on live traffic once (household-8/$8,500), not just in mocks, closing the "proven in isolation but not yet in production" gap noted after the first closure attempt.
  - **General principle:** when a fix targets one detected failure *signature*, also add an unconditional "never do the definitely-wrong thing" check as a final backstop — signature-specific detection is valuable for triggering a *smart* recovery, but it can only catch variants of the problem it was explicitly built to recognize, and this project has now twice found a variant it didn't anticipate. The two layers serve different purposes and neither replaces the other.

- **Distance metric verification:** Before interpreting ChromaDB distance scores, confirm the collection's actual `hnsw:space` metadata (cosine vs. L2/Euclidean default) rather than assuming from score magnitude alone. Check via `collection.metadata` on the live collection object.
  - **Confirmed for this project:** `ingest.py` creates the collection without an explicit `hnsw:space` override, so ChromaDB defaults to **squared L2 (Euclidean)**, not cosine. Verified both via `collection.configuration_json` (`{'hnsw': {'space': 'l2', ...}}`) and by manually computing both distances for a known pair and confirming the returned value matches squared L2, not cosine.
  - Since `text-embedding-3-small` vectors are unit-normalized, squared L2 and cosine distance are monotonically related (`L2² = 2 · cosine_distance`), so rankings are identical either way — only the raw numeric scale differs (L2 values run ~2x cosine values). Decision: keep L2 default, don't re-ingest to switch to cosine, since it would cost a full re-embed for zero ranking difference. Document this relationship in `retrieval.py` so raw distance values aren't misread later as cosine-scale.

- **Retrieval query must reflect conversation context, not just the literal current turn (found and fixed):** `retrieval.retrieve(user_input, ...)` was originally called using only the verbatim current-turn user input as the query. This works when a turn's text carries real semantic content, but fails when a turn is short and context-dependent — e.g., a bare "no" answering "any other benefits?" retrieves based on "no"'s own meaning (nothing relevant), not on what the conversation is actually about at that point.
  - **How it was found:** an apparent inconsistency (same question type, different confidence across two conversations) was first hypothesis-tested against generic phrasing sensitivity — ruled out via a 5-query diagnostic that reliably retrieved the target chunk every time. That result meant the cause was more specific than phrasing, so direct instrumentation (temporary logging of the actual retrieval query and results per turn) was added to trace the real mechanism rather than guessing further.
  - **Confirmed root cause:** turn 4 of a 4-turn sequence ("2", "2000", "sorry i meant 2200", "no") retrieved zero income-limit content because the query was literally just `"no"`. The model's honest "I don't have that information" response was the groundedness guardrail working correctly — the information genuinely wasn't in context for that call. This is a retrieval-layer gap, not a generation-layer honesty/consistency bug.
  - **Fix:** retrieval queries are now built from the last 2-3 turns of conversation rather than the bare current-turn text alone. This is a retrieval-input-only change — it does not add a new LLM call and does not grow the generation-time context window, so it doesn't trade groundedness discipline for retrieval coverage. Retrieval-time context and generation-time context are separate budgets; enriching one doesn't require enriching (or risking hallucination in) the other.
  - **General principle:** when an apparent bug could be either "retrieval problem" or "generation problem," test retrieval directly and in isolation first (diagnostic queries, direct `retrieve()` calls) before assuming which layer is at fault. Don't patch generation-layer phrasing to work around what might actually be a retrieval-layer gap.

- **Follow-on: verbose-turn dilution, four query-construction variants tested, resolved by widening top_k (not query construction):** A related symptom surfaced after the above fix — a different conversation missed retrieval entirely when a long, topically-broad assistant reply (an appeals-process tangent) sat in the recent-history window. Ablation-tested four query-construction approaches (bare current turn / full history / facts-only / facts+last-turn) across two real scenarios — no single formula won both; short-answer cases need more context folded in, verbose-tangent cases need less dilution, and these pull in opposite directions for any fixed concatenation scheme.
  - **Resolution:** rather than continuing to search for the perfect query formula, widened `top_k` from 5 to 8 (query construction unchanged from the original fix above). The "missing" target chunk in the dilution case wasn't actually absent — it ranked 7th, just outside a 5-wide window. At `top_k=8` it's captured cleanly with zero irrelevant noise in the surrounding results. Also confirmed all 8 chunks are forwarded into `<retrieved_evidence>` with no separate downstream truncation silently capping this lower.
  - **Verified via 5 fresh, live, non-replayed conversations** (not reused query strings): original short-answer bug, verbose-tangent dilution case, a 4-turn consecutive-short-answer stress test, and a single verbose multi-fact user turn — all passed with correctly cited, source-verified figures.
  - **Confirmed remaining gap, NOT fixed, documented as a known limitation:** a genuine topic shift (e.g., asking about deductions after several turns of income-limit discussion) can still fail retrieval even at `top_k=8`, because same-topic accumulated history dominates the query embedding over a short, on-topic current turn. Tried one cheap mitigation (repeating the current turn's text to weight it more heavily) — retested twice, did not close the gap either time. **The groundedness guardrail correctly held up under this failure in every test** — the agent stated it didn't have exact figures rather than fabricating them. Given the ~20 hour scope, accepted this as a documented limitation (system fails safely) rather than pursuing a third fix (e.g., explicit topic-shift detection, which would be the right approach at production scale).

- **Documentation described a feature as implemented and verified before it had actually been built — caught during pre-push regression testing, not before.** SPEC.md Section 21 and a README.md paragraph described a 2000-character input-length guard in specific, confident, past-tense detail ("Verified: inputs under, at, and over the boundary tested explicitly; confirmed rejected inputs don't reach `send_message()`...") — but a `git status`/grep across `src/` during full pre-push regression testing found zero code implementing any such check anywhere in the codebase. The documentation was written describing intended/planned work as if it were already done and verified, when neither had actually happened yet.
  - **Resolution:** the feature was then actually built (`cli._check_input_length()` in `cli.py`, a stateless 2000-char cap with exactly-2000-passes as the boundary decision) and genuinely verified (monkeypatched `cli.run()` harness covering under/at/over-boundary cases, rejection-doesn't-corrupt-session-state, concurrent-thread independence, and a live spot-check confirming zero interaction with the persona test suite) — making the documentation retroactively true rather than reverting the false claim. Both options were legitimate; this one was chosen since the described feature was cheap, well-specified, and worth having either way.
  - **General principle:** write verification claims ("Verified: ...", "Confirmed: ...") only after the corresponding implementation and testing have actually happened in the same working session — never draft them in anticipation of work that's intended but not yet done, even when the intended work seems trivial or certain to happen. A reader (including a future session) cannot distinguish a premature claim from a true one just by reading confident, specific prose — the only way this was caught here was an independent code-level check (`grep`/`git status`) that happened to be run before push, not anything in the documentation's own tone or detail level tipping it off as unverified.
